# Significant Decision Log

| Decision | Options Considered | Why we chose it |
|----------|--------------------|-----------------|
| **Database choice** | PostgreSQL vs SQLite | We used **SQLite** (`db.sqlite3`). The assignment gave us 2 days. SQLite requires zero configuration, is built into Django natively, and perfectly handles the relational requirements for a 6-person flat share without the deployment overhead of spinning up a Postgres container. |
| **Handling Negative Amounts** | 1. Reject row as typo<br>2. Log as a group refund | We chose **Option 2 (Refund)**. If Dev logs `-30 USD` for a Parasailing refund, the math holds up perfectly. Dev's "Total Spent" goes down, meaning the group owes him less (or he owes the group more). It elegantly handles cash-back events without writing custom "Income" logic. |
| **Chronological Split Validation** | 1. Trust the CSV<br>2. Filter by `joined_at`/`left_at` | We chose **Option 2**. We added timeline constraints to `GroupMember`. When Meera moves out on March 28th, she is automatically excluded from equal splits (like the April Rent) happening *after* her departure, satisfying Sam's request to not pay for past bills. |
| **USD to INR Currency** | 1. Dynamic API fetching<br>2. Static exchange rate | We chose **Option 2**. While fetching live rates from an API is technically superior, historical expenses require historical exchange rates on that exact date. For simplicity and reliability, we hardcoded a static rate (`1 USD = 83 INR`) in the importer script. |
| **Import User Experience** | 1. Fail silently<br>2. Crash on error<br>3. Interactive Approval | We chose **Option 3**. The rubric explicitly states "A crashed import and a silent guess are both failing answers." By building a "Review Anomalies" modal, we surface issues (like duplicate rows or missing payers) back to the user, satisfying Meera's request to "approve anything the app deletes or changes." |
